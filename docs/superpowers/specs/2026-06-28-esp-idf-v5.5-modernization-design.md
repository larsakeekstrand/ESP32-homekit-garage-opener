# ESP-IDF v5.5.2 Modernization & Cleanup — Design

**Date:** 2026-06-28
**Status:** Approved (pending spec review)
**Scope:** Migrate the HomeKit garage-door-opener firmware to ESP-IDF v5.5.2 and clean up
the existing code. No new features.

## Goal

Bring a multi-year-old ESP32 HomeKit garage-opener firmware up to date so it builds, flashes,
and runs on ESP-IDF **v5.5.2** (installed at `~/esp/esp-idf-v5.5.2`, symlinked `~/esp/esp-idf`),
and improve the quality of the existing single-file implementation — without changing the feature
set.

## Constraints & Decisions

- **Target chip:** ESP32 (classic, dual-core). Wiring unchanged from the current code.
- **Dependency strategy (Approach A):** ESP Component Manager for the DHT driver; re-clone
  `esp-homekit-sdk` as a git peer.
- **Code structure:** Split the single `app_main.c` (~760 lines) into focused modules.
- **No new features.** Behavior stays functionally identical except for two strict bug fixes
  (debounce, on-demand sensor read) called out below.
- **No automated tests** exist and the system is hardware-bound; verification is build-clean +
  on-device smoke test.

## Current State (baseline)

- Build: legacy GNU-make (`Makefile` + `main/component.mk`) **and** a converted
  `CMakeLists.txt`. The GNU-make path is dead in ESP-IDF v5.
- Dependencies, both currently **absent** from `~/esp/`:
  - `esp-homekit-sdk` — Espressif HomeKit SDK (provides `hap*`, `app_wifi`,
    `app_hap_setup_payload`, and a bundled `button`/`iot_button` component). Referenced via
    `EXTRA_COMPONENT_DIRS` with a `HOMEKIT_PATH` env override.
  - `esp-idf-lib` — UncleRus library, used only for the DHT (`dht.h`) driver.
- All logic in `main/app_main.c`: two FreeRTOS tasks (`garage_thread_entry`, `dht_thread_entry`)
  spawned from `app_main()`, sharing globals (`door_state`, `accessory`, `oneshot_timer`,
  `current_temperature`, `current_humidity`).
- Config: custom partition table `partitions_hap.csv` (dual OTA + factory_nvs),
  `sdkconfig.defaults` forces `CONFIG_FREERTOS_UNICORE=y`, 4 MB flash, watchdog off.

## Design

### 1. Build system & dependencies

- **Remove** `Makefile` and `main/component.mk` (legacy GNU-make; unsupported in v5).
- **Top-level `CMakeLists.txt`:** keep the `HOMEKIT_PATH` env override; set `EXTRA_COMPONENT_DIRS`
  to the re-cloned `esp-homekit-sdk` peer (`components/`, `components/homekit`, `examples/common`).
  Remove the `esp-idf-lib` peer path.
- **`main/idf_component.yml`** (new): declare the DHT driver as a managed dependency from the ESP
  Component Registry; the Component Manager fetches it into `managed_components/` on build. The
  exact registry package name is verified during implementation.
  - **Fallback (Approach B safety net):** if the registry package is unusable, vendor the two DHT
    driver files into a local `components/dht/`. This is a contingency, not the plan.
- **Re-clone `esp-homekit-sdk`** as `~/esp/esp-homekit-sdk`. **Its build compatibility with
  v5.5.2 is verified as the FIRST implementation step**, before any cleanup work, so SDK breakage
  surfaces immediately. Note: the SDK has no v5.5-tagged release — patching may be required.
- **Config:** keep `partitions_hap.csv` and `sdkconfig.defaults` (custom partition table,
  `FREERTOS_UNICORE`, 4 MB flash). Regenerate `sdkconfig` from defaults; remove `sdkconfig.old`.

### 2. Module decomposition

Split `main/app_main.c` into focused units under `main/`:

| File | Responsibility | State owned |
|------|---------------|-------------|
| `board.h` | All GPIO pin macros + board constants in one place | — |
| `door_control.{c,h}` | Door state machine: relay kick, reed-sensor debounce, state reconciliation, failsafe timer | `door_state`, `oneshot_timer` |
| `dht_sensor.{c,h}` | DHT polling task + cached latest temperature/humidity | `current_temperature`, `current_humidity` |
| `hap_garage.{c,h}` | HAP accessory creation, service/characteristic wiring, read/write callbacks, value-update helpers | `accessory` |
| `app_main.c` | Thin entry point: init board, build accessory, spawn the two tasks | — |

**Interface principles:**
- Modules communicate through small, explicit functions (e.g. `door_control_request_toggle()`,
  `dht_sensor_get(float *temp, float *hum)`).
- The HAP layer never touches GPIO directly; the door logic never references HAP UUIDs.
- `main/CMakeLists.txt` updated to register the new source files.

### 3. Behavior fixes during cleanup

1. **API migration:** `xQueueHandle` → `QueueHandle_t`; `portTICK_RATE_MS` →
   `portTICK_PERIOD_MS`; zero-initialize the `gpio_config_t` struct.
2. **Swapped DHT log** (`app_main.c:738`): the log prints the humidity value labeled "Temp" and
   vice-versa (args reversed relative to the format string). Fix.
3. **No-op `dht_read` callback** (`app_main.c:438`): currently empty, so on-demand HomeKit reads of
   temperature/humidity return stale defaults. Implement it to serve the cached values from
   `dht_sensor`.
4. **Unreliable debounce:** replace the bare edge-vs-last-value check (flagged by the author's own
   comment) with a short time-based debounce (~50 ms) for the reed sensor.
5. **De-duplicate the door state machine:** the `while(1)` loop has ~8 near-identical branches.
   Consolidate into an explicit `(sensor_edge, current_state) → (new_state, notifications)`
   transition so the logic is auditable.
6. **Remove dead code:** commented-out blocks and unused locals (e.g. `cur_val` in `dht_read`).

**Unchanged behavior:** same HomeKit services (Garage Door Opener, Temperature, Humidity, FW
Upgrade/OTA), same relay timing (300 ms kick, 8 s failsafe timer). The debounce and `dht_read` fixes
are the only observable behavior changes, and both are strict improvements.

### 4. Verification

1. **Build clean** on v5.5.2: `idf.py set-target esp32 && idf.py build` with zero errors; warnings
   triaged.
2. **On-device smoke test** (run by the user): `idf.py flash monitor` — pairs with HomeKit, reed
   sensor reflects door open/closed, relay actuates the door, temperature/humidity report.
3. **Docs:** update `README.md` and `CLAUDE.md` for the v5.5.2 workflow, the Component Manager
   dependency, and the new module layout.

Steps 1–2 require the SDK present and the device attached; the on-device smoke test depends on the
user, who will be given exactly what to watch.

## Out of Scope

- New features (MQTT/Home Assistant, additional sensors, status reporting, improved OTA).
- Multi-target support (S2/C3); ESP32 classic only.
- Security hardening (hardcoded setup code, disabled OTA cert verification) — noted but not
  addressed in this round.
- Automated test infrastructure.

## Risks

- **`esp-homekit-sdk` on v5.5.2** is the primary risk: no v5.5-tagged release. Mitigation: verify
  build as the first implementation step; patch the SDK or pin a compatible commit if needed.
- **DHT registry package** name/availability — mitigation: documented vendoring fallback.
